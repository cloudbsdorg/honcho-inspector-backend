What was wrong (root cause, more nuanced than the task spec):
==========================================================

The task spec said "Spring properties don't propagate to Logback — use
@DynamicPropertySource or System.setProperty in @BeforeAll". The actual
problem is that Spring Boot's LogbackLoggingSystem is a JVM-wide SINGLETON
(loaded once via SpringFactoriesLoader), and so is the underlying
ch.qos.logback.classic.LoggerContext.

Consequence: the very first @SpringBootTest to load a Spring context
initializes the FILE_JSONL RollingFileAppender against the system
properties in effect at THAT moment. Subsequent @SpringBootTest classes
do NOT re-read logback-spring.xml — they share the same LoggerContext
that was already configured.

This means:
1. Setting `System.setProperty("HONCHO_CONFIG_DIR", ...)` in @BeforeAll
   is TOO LATE — the appender was already configured with the
   previous class's value (or with the default `${HONCHO_CONFIG_DIR:-.}` = ".").
2. Setting it in a `static {}` initializer is also too late — the
   class isn't loaded until JUnit scans for it, but the first
   @SpringBootTest has already triggered Logback init.
3. `@DynamicPropertySource` only sets Spring Environment properties,
   which Logback does not read — so that doesn't help either.

The ONLY way to get per-class isolation is to FORCE the LoggerContext
to re-read logback-spring.xml after setting the system property. The
helper does this via JoranConfigurator.doConfigure() on a reset context.

What was changed
==========================================================

1. LogbackTestSupport.java — rewrote with two helpers:
   - prepareLogsDir(Path) — creates <configDir>/logs/
   - reloadConfig() — re-reads logback-spring.xml into the existing
     LoggerContext after context.reset(). Must be called AFTER setting
     the system property and BEFORE the first log event.

2. All 4 logging tests (JsonlFormatTest, LogMdcTest, LogRotationTest,
   LogScrubbingTest) follow the same pattern:
   - @BeforeAll: set HONCHO_CONFIG_DIR (and HONCHO_LOG_MAX_FILE_SIZE,
     HONCHO_LOG_TOTAL_SIZE_CAP for LogRotationTest) as System properties
   - @BeforeAll: prepareLogsDir(CONFIG_DIR) to ensure the dir exists
   - @BeforeAll: reloadConfig() to force Logback to re-read the config
   - Tests assert against CONFIG_DIR.resolve("logs/honcho-inspector.jsonl")
   - @AfterAll: clear System properties + delete the test config dir

3. LogbackConfigTest.java — UNCHANGED (it doesn't depend on file output,
   just appender registration; it still passes).

4. LogRotationTest was changed to use HONCHO_LOG_TOTAL_SIZE_CAP=10KB
   instead of HONCHO_LOG_MAX_HISTORY=2. Rationale: maxHistory is
   TIME-BASED and won't prune same-day rotations, so it cannot bound
   the file count for a sub-second test that runs thousands of
   rollovers. totalSizeCap is the only knob that prunes size-driven
   rotation in the same day. The assertion was changed from
   `total <= 3` to `totalSizeBytes <= 15KB` to verify pruning.
